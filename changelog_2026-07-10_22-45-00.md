## (a) Callback-payload redaction — FIXED
`src/hermes/ops/security_monitor.py` `_trigger_alert`:
- Previously fired callbacks with the RAW alert dict (secrets could leak to
  Slack/Discord/webhook sinks).
- Now redacts via `_redact_sensitive_data(_deep_copy_redact(data))` BEFORE any
  callback runs:
  - `_redact_sensitive_data` — top-level sensitive keys (api_key, session,
    auth, secret, token, ...) → `***REDACTED***`.
  - `_deep_copy_redact` (new, module-level) — recursively copies the payload and
    redacts value-shaped secrets (e.g. `Bearer ...`, `api_key=...`, or any
    >=16-char opaque token) and nested dicts (e.g. `headers.authorization`).
- Caller's original payload is NOT mutated (deep copy).
- Test added: `test_alert_callback_payload_redaction` (25th test) asserts the
  sink receives redacted data while the original stays intact.

## (b) tz-aware datetime sweep — DONE (verified clean)
Repo-wide scan of `src/hermes` (excl. web/dashboard): only 2 naive
`datetime.now()` calls remained, both in `src/hermes/transport/parquet_writer.py`
(filename timestamps, no tz comparison). Changed to `datetime.now(timezone.utc)`.
All other call sites already used `tz=timezone.utc`. No behavioral risk.

## (c) Required security gate — ADDED
- `scripts/security_gate.sh`: runs (1) the 25 security tests and (2) a
  `detect-secrets` audit over owned source, exiting non-zero if either fails.
  Scans `src/` excluding `.venv`, `tests/`, `dashboard/`, `src/hermes/web/`,
  `*.parquet`, `*.lock`. Generates `.secrets.baseline` on first run, then
  enforces (audit fails on NEW secrets).
- Cron job `security-gate` created: daily 03:15 PT, runs the gate, delivers
  results locally, FAIL exit alerts (per cron watchdog pattern).

## Regression avoided
A bug was introduced mid-work: `_deep_copy_redact` was first placed at module
level INSIDE the class body (col 0), which prematurely closed the
`SecurityMonitor` class (CSRF methods dropped) and was masked by a STALE
`__pycache__` `.pyc` (the run loaded old bytecode). Fixed by moving the function
to end-of-file and clearing `.pyc`. Re-verified: class intact, 25/25 tests pass.

## Additional fixes found during gate verification
- **`threading.Timer` non-daemon hang (real shutdown bug)** — `security_monitor._check_escalation` used `threading.Timer(...).start()` (non-daemon). When an escalation fired during a test, the timer thread kept the interpreter alive → **pytest hung on exit** (every run timed out at shutdown, not just the gate). Fixed: assign to var, set `daemon = True`, then start. This also prevents the production monitor from blocking clean process exit.
- **`detect-secrets` CLI path** — `python -m detect-secrets` is not exposed; use the console script `.venv/Scripts/detect-secrets.exe`. `audit` is interactive (prompts) and `--audit` is not a 1.5.0 flag, so enforcement is done by **diffing a fresh `scan` against `.secrets.baseline`** (normalized to `results`-only to ignore volatile `generated_at`/`plugins_used` metadata). Helper `scripts/_normalize_baseline.py` does the normalization.
- **MSYS path handling** — `REPO_ROOT` computed via `pwd -W` (Windows-native `C:/...`); `/tmp` is unreliable for redirects under MSYS, so the scan temp is repo-local (`.gate_ds_new.json`).
