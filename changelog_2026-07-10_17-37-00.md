**Date/Time:** 2026-07-10 17:37:00 PDT (UTC 2026-07-11T00:37:00Z)
**Agent:** Quant (noble-agent)
**Scope:** (a) Harden `scripts/watchdog.sh` so the stack auto-recovers on

## A. Watchdog hardening (`scripts/watchdog.sh`)

**Root causes found during the session (all now fixed):**

1. **`nohup ... &` children died on session-teardown SIGTERM.** The original
   launcher could not survive a system sleep. → Switched to PowerShell
   `Start-Process` (the only method on this git-bash/Windows host that detaches
   into a new independent process tree).
2. **`-RedirectStandardOutput` and `-RedirectStandardError` pointed at the SAME
   file** → `Start-Process` threw `InvalidOperationException` (exit 1) and
   spawned NOTHING. This was the silent "0 processes" failure. → stderr now goes
   to a separate `<name>.err` file; JSON stream stays in the main `<name>.log`.
3. **`-PassThru` with redirection blocked** PowerShell on the stream handle
   until the process exited (deadlock for servers). → dropped `-PassThru`; PID is
   captured via a follow-up `Get-CimInstance` query instead.
4. **Venv python was passed BOTH as `-FilePath` AND as the first `-ArgumentList`
   element** → ran `python.exe python.exe -m ...` → crash. → venv path is only
   the `-FilePath`; args array starts at `-m`.
5. **`.bat` wrapper did not survive** detachment. → removed; python launched
   directly.
6. **`proc_alive` was PID-file / single-PID based** → stale-PID false-negatives
   (and the uvicorn reloader spawns a child worker, so the "parent" PID could die
   while the loop stays up). → liveness is now **NAME-BASED** (`Get-CimInstance`
   for any live python running our loop). Robust against reloader reshuffle.
7. **No single-instance guard** → concurrent runs (cron tick + manual, or two
   overlapping ticks) double-launched. → added a `.watchdog.lock` with live bash
   PID; a second instance exits immediately if the first is still alive.
8. **Silent no-op on launch failure.** → every launch is now verified 4s later;
   a failure logs a hard `FAIL — <loop> did not come up` to `/tmp/watchdog.log`.

**Verified behavior:**
- Single clean run launches exactly **7 instances** (6 loops + 1 optimizer
  watcher), each = 1 venv parent + 1 uvicorn-reloader child worker (normal; the
  child is the actual server). No port-8080 conflict (dashboard log shows
  "Uvicorn running on http://127.0.0.1:8080" once).
- A **second run** (simulating the 5-min cron tick) detects all loops alive and
  relaunches NONE — log shows only "redis already up" + "check complete".
- Redis also launched detached + PID-guarded, restarts if killed.

**Cron:** `noble-stack-watchdog` (job 1359279cee62) resumed, `*/5 * * * *`.
The watchdog is now the single owner of the stack; do NOT also launch loops via
`terminal(background)` (that was the source of the earlier 2× duplicates).

## B. Audit-report XSS + render crash fix (`src/hermes/security/audit_logger.py`)

**Bug:** `_generate_html_report()` used `html_template.format(...)` on a template
that contains CSS braces (`body { font-family: Arial, sans-serif; ... }`).
`str.format()` parses `{` as a field marker → `KeyError` the moment the report
is rendered. (The earlier-committed XSS escaping of user fields was correct, but
the generator itself was broken by the `.format()`/CSS collision.)

**Fix:** Switched the template to `string.Template` (uses `$generated`,
`$total_events`, `$rows`; ignores `{`/`}`). Verified: report generates (len 1566),
CSS preserved, and a malicious `<script>alert(1)</script>` username is escaped to
`&lt;script&gt;` (raw payload absent).
