# Changelog 2026-07-11 15:55:00 — `noble config` per-venue param audit command

## Context
User wanted an easy way to view per-venue trading params (Alpaca + Hyperliquid)
from `config/default.yaml` and request audited edits, for review/audit purposes.
Confirmed config changes ARE tracked in DuckDB via `src/hermes/db/config_history.py`
(table `config_history`: config_hash, ts, config_json, source, rationale, author,
diff) with `apply_config_change` / `get_config_history` / `get_config_by_hash`.

## Changes
### src/hermes/commands/noble_cli.py
- Added `noble config` Click command to the existing `noble` group:
  - Lists every per-venue param (alpaca + hyperliquid) from `config/default.yaml`.
    Credentials rendered as `secret:***` (values never exposed).
  - Also prints the ACCOUNT-LEVEL RISK block (max_leverage_total,
    max_gross_exposure_pct, max_portfolio_drawdown_pct, daily_loss_limit_pct)
    and the daily_wins / daily_profit cooloff circuit breakers — the real
    leverage / exposure / kill-switch controls.
  - `--venue alpaca|hyperliquid` — narrow to one venue.
  - `--key <dotted.path>` — show only one param.
  - `--audit` — for each key, prints the DuckDB `config_history` trail (last
    change ts / author / source / rationale); also reports whether the current
    snapshot hash is already recorded.
  - `--set 'KEY=VALUE' --why '<rationale>'` — request a tracked change via
    `config_history.apply_config_change` (writes the DuckDB row BEFORE editing
    the YAML file). Requires `--why`; tier-locked keys still follow autonomy
    tier_3/tier_4 rules.

### skills/trading/noble/SKILL.md
- Documented `noble config` (all subcommands + audit + change-request flow) in
  the Usage section and a new "`noble config` — per-venue param audit" block.
