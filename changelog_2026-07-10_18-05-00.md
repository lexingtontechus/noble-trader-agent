## Root cause (the file was "silently uncollectible")

Two SyntaxErrors, one in the test file and one in the imported source module:

1. **`tests/test_security_scenarios.py:65`** — `positional argument follows keyword
   argument`:
   ```python
   _sanitize_input(long_input, max_length=100, "username")   # BAD
   ```
   Fixed → `_sanitize_input(long_input, max_length=100, field_name="username")`.

2. **`src/hermes/ops/security_monitor.py:1090`** — THIS was the real blocker. A
   corrupted line made the entire module unimportable, so pytest errored on
   collection (not a clean "uncollectible", but the import failure hides all 24
   tests):
   ```python
   error=str(e"          # BAD — unterminated string literal
   ```
   Fixed → `error=str(e)`.

## Additional source defects found during verification (fixed)

3. **`_sanitize_input` signature mismatch** (`security_monitor.py:1529`) — the test
   and other callers pass `field_name` as the 2nd positional argument, but the
   signature was `(value, max_length=1000, field_name="input")`, so `"username"`
   bound to `max_length` → `TypeError: '>' not supported between int and str`.
   Reordered to `(value, field_name="input", max_length=1000)`. Internal callers
   use `field_name=` as a keyword, so they are unaffected.

4. **`RateLimiter` naive/aware datetime crash** (`security_monitor.py:167,173,185`)
   — `datetime.now()` (naive) compared against tz-aware datetimes raised
   `TypeError: can't compare offset-naive and offset-aware datetimes`. Changed all
   three to `datetime.now(timezone.utc)`.

## Update (2026-07-10 18:09 PDT): redaction hardened (user priority)

`test_data_redaction_sensitive_fields` was failing because `_redact_sensitive_data`
used EXACT field-name matching (`key` matched `key` but not `api_key` /
`session_id`). Rewrote to **substring token matching**: any field whose
(lowercased) name contains a sensitive token (`password`, `pass`, `token`,
`secret`, `key`, `authorization`, `auth`, `cookie`, `session`, `credential`,
`private`, `api_key`) is redacted. This is the fail-safe blacklist pattern used by
structlog/blacklist redactors — it catches `api_key`, `access_token`,
`session_id`, `secret_key`, `auth_token`, etc. while leaving `username` and
`normal_field` untouched. Test now passes. 10/24 pass; 14 remain assertion
mismatches.
