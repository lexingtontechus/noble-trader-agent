## Fixes applied

1. **Test syntax** (`tests/test_security_scenarios.py:65`):
   `_sanitize_input(long_input, max_length=100, "username")` →
   `max_length=100, field_name="username")`.

2. **Source SyntaxError (the real blocker)** (`security_monitor.py:1090`):
   `error=str(e"` → `error=str(e)`. This made the whole module unimportable,
   so all 24 tests were silently uncollectible.

3. **`_sanitize_input` signature** (`security_monitor.py`): callers pass
   `field_name` as 2nd positional; signature had `max_length` 2nd → `TypeError`.
   Reordered to `(value, field_name="input", max_length=1000)`.

4. **RateLimiter naive/aware datetime** (`security_monitor.py`): `datetime.now()`
   → `datetime.now(timezone.utc)` (3 sites) — fixed `TypeError` on comparison.

5. **`api_rate_limit` decorator recursion bug** (`security_monitor.py:62`):
   `type(self).log_security_event.__wrapped__` raised `AttributeError` (no
   `__wrapped__` on the class-bound method), crashing every security-event log.
   Fixed to use the captured `func` closure (the undecorated original).

6. **HTML/script tag stripping in `_sanitize_input`** (`security_monitor.py`):
   added `re.sub(r'<[^>]*>', '', result)` — defense-in-depth XSS/injection control
   so `<script>alert(1)</script>` becomes `alert(1)`. (Test
   `test_xss_attack_attempt` + control-char test now pass.)

7. **Redaction hardening** (`security_monitor.py` `_redact_sensitive_data`):
   the source already did substring matching on `sensitive_tokens`
   (api_key, session, auth, secret, token, …) → `api_key`/`session_id` now
   correctly redact to `***REDACTED***`. (Test `test_data_redaction_sensitive_fields`
   passes.) Defensive redaction-before-callback is still advisable (see note).

8. **CSRF one-time-use session model** (`security_monitor.py` `validate_csrf_token`):
   was per-token removal, leaving prior tokens replayable. Now consuming ANY
   token invalidates the entire session's token set → closes CSRF replay/hijack.
   (Test `test_penetration_test_session_hijacking` passes.)

9. **Test expectation correction** (`tests/test_security_scenarios.py`
   `test_input_sanitization_special_characters`): the test asserted `\xff` (which
   in a Python `str` is the printable char `ÿ`, U+00FF) be stripped — impossible
   alongside its own assertions that `ü`/`ä`/`😊` (also printable Unicode) be KEPT.
   Aligned the test: control chars (`\x00\x01\x02`) stripped, printable `\xff`
   preserved. Source sanitizer behavior is correct & secure; the assertion was
   self-contradictory.

## Residual notes (not bugs, but hardening recommendations)
- `_trigger_alert` fires callbacks with the raw alert dict. For defense-in-depth,
  redact sensitive fields in the alert payload BEFORE invoking callbacks (so a
  Slack/Discord/webhook sink never receives a raw token). Low-risk addition.
- `RateLimiter` still mixes `datetime.now(timezone.utc)` only in the 3 edited
  sites; other `datetime.now()` calls elsewhere in the file remain naive. If any
  compare against tz-aware values they'd resurface — recommend a repo-wide
  `datetime.now(timezone.utc)` sweep.
