# Changelog 2026-07-11 15:45:00 — config tuning + daily-wins/profit kill switch

## Context
User review of `config/default.yaml` (per-venue trading params). Concerns:
- HL `max_leverage=50` (from HL API) is dangerously high and "will blow out any account".
- Need a "daily wins" circuit breaker / kill switch: if the account runs hot vs
  its avg daily wins / total profit, halt trading for the day/session (cooloff)
  and ride open positions to their exits before a likely regime shift.

## Findings (verified from code, not assumed)
- **Leverage is NOT used for sizing.** `signals/sizing.py` caps size by
  max_position_pct / max_notional / max_gross_exposure / risk_amount only. The
  per-symbol `max_leverage` pulled from HL's API (40-50x) is stored in the
  symbols table but never feeds sizing. So a 50x blowout path does NOT execute.
  The enforced controls are `account.max_leverage_total=3.0`,
  `asset.max_position_notional=$25k`, `account.max_gross_exposure_pct=1.50`.
- The config ALREADY had a `circuit_breakers.manager.daily_wins` block, but
  `cb_manager.py` had **no `check_daily_wins` method** — it was configured but
  never evaluated (dead config). There was also no profit-cap variant.

## Changes
### config/default.yaml
- `venues.hyperliquid.features.max_leverage`: 10.0 → **5.0** (reference ceiling;
  enforced cap remains `account.max_leverage_total: 3.0`). Comment clarifies the
  per-symbol HL 40-50x value is intentionally NOT used for sizing.
- Added `circuit_breakers.manager.daily_profit` (profit-cap cooloff, mirrors
  daily_loss but fires on realized PROFIT): $3k reduce_50pct / $5k block_entries
  (4h) / $8k halt_all (8h). Open positions are never force-closed — they ride to
  their own TP/stop.
- Kept `daily_wins` (wins_today/avg_daily_wins ratio): 1.5× reduce / 2.0× block
  (4h) / 3.0× halt (8h).

### src/hermes/portfolio/cb_manager.py
- Added `check_daily_wins(wins_today, avg_daily_wins)` → ratio vs tiered config;
  no-ops when avg_daily_wins<=0 (cold start safe).
- Added `check_daily_profit(daily_profit_usd)` → tiered profit cooloff; no-ops
  when profit<=0.

### src/hermes/portfolio/state.py
- Added `get_wins_today()` and `get_avg_daily_wins()` accessors (back the
  daily_wins CB).
- Fixed hardcoded margin assumption `gross/4.0` → `gross/3.0` to reflect the
  enforced `max_leverage_total` cap (reporting only; sizing unaffected).

### src/hermes/portfolio/orchestrator.py (`_evaluate_risk`)
- Wired `check_daily_wins` + `check_daily_profit` into the per-signal CB sweep,
  alongside the existing `check_daily_loss`. Trips now apply the size multiplier
  / block / ride-and-exit exactly like the other categories.

## Verification
- `py_compile` on all 3 modified .py files: OK.
- Behavior test via `CircuitBreakerManager.from_config`:
  - wins 6 / avg 3 (2.0×) → `block_entries` + 14400s cooloff ✓
  - profit $6k → `block_entries` + 14400s cooloff ✓
  - profit $1k (under $3k) → 0 trips ✓
  - cold start (avg=0) → 0 trips (no false fire) ✓

## Notes
- `block_entries` / `halt_all` from these CBs block NEW entries only; existing
  open positions exit on their own TP/stop (matches "ride and exit" intent).
- These config keys (`daily_wins`, `daily_profit` under circuit_breakers.manager)
  are NOT in autonomy tier_3/tier_4 locked lists, so changing tiers via the
  existing auto-promotion path follows normal autonomy rules. The top-level
  `circuit_breakers.enabled` / `kill_switch.auto_triggers` remain tier-4 locked.
